GlobleMind Engineering Reference

Document Identity, Versioning & Deduplication

How GlobleMind decides whether a file is new, a duplicate, or a replacement — and what happens at every step from upload through embedding to the frontend Documents page.

July 2026 10 pipeline stages Content-addressed identity

Three Guarantees

Every design decision flows from three invariants the system never violates:

1
Deduplication
Identity is the SHA-256 of the file’s content, never its name. Re-uploading identical bytes under any filename is a no-op.
2
No Data Loss
Two different files that share a name (two people’s resume.pdf) are two independent documents. Both are kept.
3
Safe Replacement
“Replace X with Y” creates a new version and supersedes the old one. The old version stays live until the new one is fully indexed.
The core rule: identity is content, not filename. Two files named resume.pdf with different bytes are two separate documents. Two files named a.pdf and b.pdf with identical bytes are the same document.

Ingestion Pipeline

Every uploaded file passes through 10 stages. The frontend streams progress in real time as a step-by-step card in the active chat.

Pipeline Stages (in order)
1. File Detection
MIME type, encoding, file category (PDF, DOCX, text, image…)
2. Document Classification
Structural type (text-heavy, tabular, scanned) + semantic type (report, invoice, legal…)
3. Content Parsing
Extract raw text and page structure from the document format
4. OCR
Optical character recognition for scanned pages (non-fatal if provider unavailable)
5. Layout Analysis
Detect headers, columns, reading order (non-fatal)
6. Table Extraction
Structured table data from embedded tables (non-fatal)
7. Visual Analysis
Describe charts, diagrams, images via vision model (non-fatal)
8. Chunking
Split parsed content into retrieval-sized chunks with metadata
9. Embedding
Dense + sparse vector embeddings for each chunk
10. Vector Store
Upsert chunks with embeddings into Qdrant, all marked active: true

Stages 4–7 depend on external AI providers (vision, OCR) that may rate-limit or fail on free tiers. Each is non-fatal: a failure logs a warning and the document continues through the remaining stages with whatever text was already parsed. A document is never dropped because of a provider outage in these stages.

Upload Scenarios at a Glance

The system’s behavior depends on two axes: whether the content (bytes) matches an existing active document, and whether the filename matches. Here is every combination:

Scenario Same Content? Same Name? Outcome
Brand-new file No No Ingested as a new document
Exact duplicate Yes Yes Skipped (dedup hit)
Same content, different name Yes No Skipped (dedup hit)
Different content, same name No Yes Ingested as a separate new document
Replace (via UI button) No Any New version; old one superseded
Replace with identical content Yes Any Skipped (dedup hit, no-op)
Re-upload superseded content Matches old version Any Ingested as a new document (inactive content doesn’t block dedup)

Replace 5. Replace a Document (via the Replace button)

The user clicks Replace on quarterly-report.pdf (v1) and picks a new file. The old version stays fully searchable while the new one is being processed.

What happens (atomic cutover)

Step 1New file is ingested through all 10 stages. Chunks born active: true.
Step 2New version registered in metadata: active: true, supersedes: doc-v1.
Step 3Old chunks in Qdrant marked active: false (hidden from search).
Step 4Old version metadata updated: active: false, superseded_by: doc-v2.
Safety guarantee: If the pipeline crashes before step 2, the old version is untouched (fully active). If it crashes between steps 2 and 4, both versions are temporarily active — a harmless duplicate, never zero active versions.

Delete 6. Delete a Document

The user clicks Delete and confirms. The document is removed from both the vector store and the registry.

What happens

Vector storeAll chunks belonging to this document_id are deleted from Qdrant.
RegistryEntry removed. If it had superseded an older version, that version is reactivated.

If you delete version 2, version 1 comes back as the active version — the lineage is never left with nothing active.

New Document 7. Re-upload Previously Superseded Content

Document v1 was superseded by v2. Now someone uploads a file with v1’s exact bytes again. The dedup check only looks at active documents, so the inactive v1 doesn’t count as a match.

What happens

Registry checkSHA-256 matches old v1, but v1 is active: false → no active match → NEW_FILE
PipelineFull pipeline runs. Content is re-embedded and stored.
RegistrationNew document with its own document_id and its own lineage.

This is intentional: superseded content should be re-ingestable. The user explicitly replaced it; re-uploading it means they want it back as a new, independent document.

Where the Data Lives

All durable state lives in Qdrant — a single source of truth that survives any restart, redeploy, or migration to another host.

Durable Storage Architecture
Chunk embeddings Qdrant collection globle_mind — dense + sparse vectors, payload includes active flag
Document metadata Qdrant collection globle_mind_documents — version state, lineage, dedup hashes
Local JSON fallback data/ingested_files.json — development only, when Qdrant is not configured
Upload files data/uploads/<token>/filename — transient; content already lives in Qdrant embeddings
Chat history data/ — local only (ephemeral on free PaaS tiers)

Startup Reconciliation

When the app starts and Qdrant is configured, a reconciliation pass runs automatically. It forces every superseded document’s chunks to active: false in the vector store, healing any drift from a crash mid-cutover. This is idempotent and best-effort — a failure never blocks startup.

Legacy Migration

Existing installations that used the old SHA-256–keyed JSON registry are migrated automatically on first startup. The migration rekeyes entries by document_id, adds lineage fields (active, lineage_root, supersedes, superseded_by), and normalizes field names (file_name → filename, ingested_at → created_at). When Qdrant is configured, the JSON data is seeded into Qdrant once, then the JSON file serves only as a local cache.

Search & Retrieval Filtering

Every query path — dense search, hybrid search, document-list answers — passes through a single filter function that always applies:

must_not: [active = false]
Chunks explicitly marked active: false are excluded from results. Chunks without an active field (pre-existing data) are included by default for backward compatibility.

This means superseded document versions never appear in search results, even if the supersede step completed but the chunk flags weren’t updated yet (the startup reconciliation closes that gap).

Registry Entry Schema

Each document version is stored as a single entry keyed by document_id:

Schema Document Version Entry
document_idUUID — stable identity of this specific version
content_hashSHA-256 of the file bytes — the dedup key
filenameOriginal filename (display only, never used for identity)
file_pathUpload path on disk
file_size_bytesFile size at time of ingestion
total_chunksNumber of chunks created
created_atISO 8601 timestamp
supersedesThe document_id this version replaced (null if first version)
superseded_byThe document_id that replaced this version (null if current)
activetrue for the current live version, false for superseded ones
lineage_rootThe document_id of the first version in this replacement chain
